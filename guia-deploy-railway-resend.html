Parte 1

Deploy no Railway

O Railway detecta automaticamente projetos Node.js com Dockerfile e já possui PostgreSQL nativo. Não precisa de configuração de servidor.

🚂
Railway
railway.com · Plano gratuito disponível (Hobby: $5/mês para produção)
Acessar →
Criar conta no Railway

Acesse railway.com e clique em Start a New Project. Faça login com sua conta GitHub — isso vai permitir deploy automático a cada git push.

Criar novo projeto e subir o repositório

No painel do Railway, clique em New Project → Deploy from GitHub repo. Selecione o repositório do Montador Pro.

💡 Se o projeto ainda não está no GitHub, crie um repositório e faça push da pasta backend/. O Railway detecta o Dockerfile automaticamente. 
# Na pasta backend do projeto:
git init
git add .
git commit -m "chore: initial commit"
git remote add origin https://github.com/SEU_USER/montador-pro-api.git
git push -u origin main
Configurar o Root Directory

Se o repositório contém a pasta backend/ e frontend/ juntos, é necessário informar ao Railway qual é a pasta do backend:

No Railway → seu serviço → Settings → Source → Root Directory
Digite: backend

Gerar domínio público

Após o build concluir, vá em:
Seu serviço → Settings → Networking → Generate Domain

Você receberá uma URL no formato:
https://montador-pro-api-production.up.railway.app

✅ Esta é a URL base da sua API. Use-a nas variáveis de ambiente do frontend (CFG.API no app.html) e nos outros serviços.
Parte 2

PostgreSQL no Railway

O Railway oferece PostgreSQL gerenciado com backups automáticos. É a forma mais simples de ter um banco em produção.

Adicionar banco de dados ao projeto

No canvas do Railway, clique em + New → Database → Add PostgreSQL.

O banco é provisionado em segundos. O Railway expõe automaticamente as variáveis:

PGHOST       = hostname do banco
PGPORT       = 5432
PGDATABASE   = railway
PGUSER       = postgres
PGPASSWORD   = gerado automaticamente
DATABASE_URL = postgresql://postgres:SENHA@HOST:PORT/railway
Conectar o banco ao serviço backend

No serviço do backend, vá em Variables → Add Variable e adicione:

DATABASE_URL = ${{Postgres.DATABASE_URL}}

A sintaxe ${{Postgres.DATABASE_URL}} é uma referência dinâmica do Railway — ele injeta automaticamente a URL correta do banco provisionado.

⚠️ O Prisma precisa da DATABASE_URL para rodar as migrations. O Dockerfile já executa prisma migrate deploy no startup — certifique-se de que a variável está configurada antes do primeiro deploy.
Verificar migrations e seed

Após o deploy, abra os logs em Deployments → View Logs e confirme que as migrations rodaram:

✓ Migrations applied successfully
✓ Seed data inserted (demo montador)
🚀 Montador Pro API rodando em http://localhost:3000

Se as migrations não rodaram, acesse o terminal do Railway (serviço → Deploy → Connect) e rode manualmente:

npx prisma migrate deploy
node prisma/seed.js
Parte 3

Variáveis de Ambiente

Configure todas as variáveis no painel do Railway em seu serviço → Variables. Clique em + New Variable para cada uma.

🔐 Nunca faça commit do arquivo .env com secrets reais. O Railway injeta tudo via painel, sem necessidade de arquivo .env em produção.
Variável Valor / Descrição Obrig.
DATABASE_URL ${{Postgres.DATABASE_URL}} — referência automática ao PostgreSQL do Railway SIM
NODE_ENV production SIM
PORT 3000 (Railway injeta automaticamente — pode omitir) AUTO
JWT_SECRET String aleatória longa. Gere com: openssl rand -hex 64 SIM
JWT_REFRESH_SECRET Outra string aleatória diferente do JWT_SECRET SIM
JWT_EXPIRES_IN 15m SIM
JWT_REFRESH_EXPIRES_IN 30d SIM
ADMIN_SECRET Senha usada no painel admin. Gere com: openssl rand -hex 32 SIM
VAPID_PUBLIC_KEY Chave pública VAPID para push notifications (já configurada no sistema) PUSH
VAPID_PRIVATE_KEY Chave privada VAPID — obrigatória se quiser push real PUSH
VAPID_SUBJECT mailto:admin@nfmoveis.com.br PUSH
RESEND_API_KEY Chave da API do Resend (começa com re_) — ver Parte 4 EMAIL
RESEND_FROM NF Móveis <noreply@nfmoveis.com.br> — domínio verificado no Resend EMAIL
CORS_ORIGIN URL do frontend, ex: https://testenf-production.up.railway.app NÃO
MAX_FOTO_MB 5 NÃO

Gerar secrets seguros (rodar localmente)

# JWT Secret
openssl rand -hex 64
# saída: a1b2c3d4e5f6... (128 chars)

# JWT Refresh Secret (diferente do anterior)
openssl rand -hex 64

# Admin Secret
openssl rand -hex 32

# Gerar novas chaves VAPID (se necessário)
npx web-push generate-vapid-keys
# Public Key: BEl62iUYgUiv...
# Private Key: abc123...
Parte 4

Configurar o Resend

O Resend é um serviço de e-mail para desenvolvedores com plano gratuito de 3.000 e-mails/mês. Usado para notificações de convite, aprovação de montador e confirmações de serviço.

✉️
Resend
resend.com · 3.000 emails/mês grátis · API simples
Acessar →
Criar conta no Resend

Acesse resend.com e crie uma conta gratuita. Você pode começar enviando pelo domínio de teste onboarding@resend.dev sem configurar um domínio próprio.

Gerar a API Key

No painel do Resend: API Keys → Create API Key

  • Name: Montador Pro Production
  • Permission: Sending access
  • Domain: All domains (ou selecione o seu)
🔴 A chave começa com re_ e aparece apenas uma vez. Copie imediatamente e salve em local seguro antes de fechar a janela.

Adicione a chave no Railway em RESEND_API_KEY.

Verificar domínio próprio (opcional mas recomendado)

Para enviar de @nfmoveis.com.br, vá em Resend → Domains → Add Domain e siga:

  1. Digite o domínio: nfmoveis.com.br
  2. O Resend mostrará registros DNS (TXT, MX, DKIM)
  3. Adicione esses registros no painel do seu registrador de domínio (Registro.br, Cloudflare, etc.)
  4. Aguarde de 5 min a 48h para propagação
  5. Clique em Verify no Resend
✅ Sem domínio verificado, use onboarding@resend.dev como remetente — funciona perfeitamente para testes e desenvolvimento.
Parte 5

Integrar Resend ao Backend

Adicionar o SDK do Resend ao backend Node.js para enviar e-mails de notificação automáticos.

1. Instalar o SDK

npm install resend

2. Criar o serviço de e-mail

Crie o arquivo backend/src/utils/email.js:

// backend/src/utils/email.js
const { Resend } = require('resend');

const resend = new Resend(process.env.RESEND_API_KEY);

const FROM = process.env.RESEND_FROM || 'NF Móveis <onboarding@resend.dev>';

// ── Email: Convite de Serviço ──────────────────────────────
async function emailConvite({ para, nomeMontador, descricao, endereco, data, valor, linkApp }) {
  return resend.emails.send({
    from: FROM,
    to: [para],
    subject: `🔔 Novo Serviço Disponível — R$ ${valor}`,
    html: `
      <div style="font-family:sans-serif;max-width:520px;margin:0 auto">
        <div style="background:#004d4d;color:#fff;padding:24px 28px;border-radius:12px 12px 0 0">
          <h2 style="margin:0;font-size:22px">🔔 Novo Serviço, ${nomeMontador.split(' ')[0]}!</h2>
        </div>
        <div style="background:#f8f7f4;padding:24px 28px;border-radius:0 0 12px 12px;border:1px solid #e5e4e0">
          <p style="font-size:28px;font-weight:800;color:#004d4d;margin:0 0 16px">R$ ${valor}</p>
          <p><strong>Serviço:</strong> ${descricao}</p>
          <p><strong>📍 Endereço:</strong> ${endereco}</p>
          <p><strong>📅 Data:</strong> ${data}</p>
          <p style="margin-top:24px">
            <a href="${linkApp}" style="background:#C5A059;color:#fff;padding:12px 24px;border-radius:8px;text-decoration:none;font-weight:700">

  });
}

// ── Email: Aprovação de Montador ──────────────────────────

// ── Email: OS Concluída (para o admin) ───────────────────
async function emailOSConcluida({ para, nomeMontador, numero, valor }) {
  return resend.emails.send({
    from: FROM,
    to: [para],
    subject: `🎉 OS ${numero} Concluída — ${nomeMontador}`,

module.exports = { emailConvite, emailAprovacao, emailOSConcluida };

3. Usar nos endpoints existentes

Adicione chamadas de e-mail nos pontos principais do backend (opcional — o sistema funciona sem e-mail, usando apenas push notifications):

// Em backend/src/routes/admin.js — ao criar convite:
const { emailConvite } = require('../utils/email');

// Dentro do POST /convites, após criar o convite:
if (process.env.RESEND_API_KEY) {
  try {
    const m = await prisma.montador.findUnique({ where: { id: montadorId } });
    if (m?.email) {
      await emailConvite({
        para: m.email,
        nomeMontador: m.nome,
        descricao: os.descricao,
        endereco: os.endereco,
        data: os.dataInstalacao.toLocaleDateString('pt-BR'),
        valor: os.valorBruto,
        linkApp: `${process.env.APP_URL || ''}/app/`,
      });
    }
  } catch(e) { /* email falhou, mas convite continua ativo */ }
}

4. Adicionar campo email no schema (opcional)

Se quiser e-mails para os montadores, adicione o campo no schema Prisma:

// Em prisma/schema.prisma — modelo Montador:
model Montador {
  // ... campos existentes ...
  email  String?   // adicionar esta linha
}
# Após editar o schema, rodar a migration:
npx prisma migrate dev --name add_email_montador
